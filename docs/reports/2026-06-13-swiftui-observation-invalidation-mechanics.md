## Observation as the dependency collector

The public Observation API is small, but the open-source Swift implementation makes its mechanics unusually visible. `generateAccessList` creates a scoped access list by installing a thread-local pointer, running the tracked closure, and then deactivating the list afterward. Each `ObservationRegistrar.access` call checks that thread-local slot, lazily creates an `_AccessList` if needed, and adds the accessed key path grouped by registrar context. In other words, read-dependency capture is not based on global subscriptions or object-wide dirty flags; it is an access-list assembled during execution of a particular scope. ŅłĆciteŅłéturn11view0Ņłéturn15view0Ņłéturn18view0Ņłéturn18view2Ņłü

Once that access list exists, `ObservationTracking._installTracking` registers callbacks for the accessed key paths. The public `withObservationTracking(_:onChange:)` overload installs a `willSet` observer and cancels the tracking after the first callback, which is why the function is effectively one-shot. The SwiftUI-only SPI overloads are more revealing: there are separate overloads for `willSet`, `didSet`, and `willSet` plus `didSet`. That API shape strongly suggests the intended integration point for a render engine that wants both immediate dirty-marking and some notion of an end-of-update phase. ŅłĆciteŅłéturn17view2Ņłéturn17view1Ņłéturn11view0Ņłü

This one-shot behavior is also visible in user-facing discussion. Developers on Swift Forums quickly discovered that `withObservationTracking` ŌĆ£only fires onceŌĆØ unless they recursively re-arm it, and the source code confirms why: the default public implementation explicitly cancels the tracking after the first change. That is exactly the sort of behavior a UI framework can hide by wrapping each render pass in a fresh observation scope, but it is awkward for ad hoc user code outside SwiftUI. ## State and local view invalidation

AppleŌĆÖs state-and-binding guidance still describes `@State` as the local source of truth for a view and as the thing you project into bindings for children. The official SwiftUI tutorial positions `@State` as local storage and `@Binding` as the way to share that storage with a subview. That is the conceptual baseline: state is owned locally, and updates should propagate to views that depend on it. ŅłĆciteŅłéturn24search12Ņłü

What Apple does not publicly spell out in detail for plain `@State` is the exact read-tracking mechanism. OpenSwiftUI fills that gap with a model that is technically plausible and consistent with SwiftUIŌĆÖs dependency semantics. In its `State` implementation, reading `wrappedValue` during an update pass calls `getValue(forReading: true)`, and if the graph host is currently updating, the state locationŌĆÖs `wasRead` flag is set. By contrast, creating the projected binding uses `getValue(forReading: false)`, which means merely forming `$state` does not mark the state as read. That distinction is critical: passing a binding down is not the same as the parent view depending on the bound value for its own `body`. ŅłĆciteŅłéturn38view0Ņłü

The real punchline appears in `StatePropertyBox.update`. OpenSwiftUI checks whether the state signal changed, refreshes the cached value from storage, and then only reports the property as changed if the location was newly installed or `newLocation.wasRead` is true. In other words, for that implementation, an unread state change does not automatically invalidate the direct owning view on subsequent passes. That is the clearest publicly inspectable evidence I found for read-elision of local state. A second nuance is local observable models. ## Binding and propagated dependencies

`@Binding` is best thought of as access plumbing, not as a source-of-truth mechanism. A succinct Swift Forums answer states the key point directly: `Binding` itself does nothing regarding view invalidation; invalidation comes from the underlying `@State` or observed model. That forum thread predates Observation, but the basic point remains correct: a binding is only as precise as the storage location behind it. ŅłĆciteŅłéturn8view1Ņłü

OpenSwiftUIŌĆÖs `Binding` implementation reflects exactly that model. Its `wrappedValue` getter calls `readValue()`, and when an update pass is underway it marks `location.wasRead = true`; its setter just calls `location.set(newValue, transaction: transaction)`. The documentation in that same file adds an important behavioral note: when a mutable binding changes, the new value is available immediately, but view updates happen asynchronously, so the UI may not show the change immediately. This matches the usual SwiftUI experience: mutation is synchronous at the model/location layer, while rendering happens later in the frameworkŌĆÖs update pipeline. ŅłĆciteŅłéturn37view3Ņłéturn38view2Ņłü

TokamakŌĆÖs `Binding` says something very similar in even plainer words. Its source comments note that `set` functions are not mutating in-place synchronous view updates; instead they schedule an update with the renderer later. Tokamak is not Observation-based, but it captures an important truth about bindings in declarative UI: a binding write is logically an upstream mutation request, not an immediate command to redraw the current view in place. ŅłĆciteŅłéturn34view3Ņłü

Putting those sources together gives a practical answer to ŌĆ£`@Binding`-based invalidation and its tracking.ŌĆØ A child holding `@Binding var value` invalidates when two things are true: the underlying storage location changes, and the childŌĆÖs render scope actually read that location. ## Observable models, `withObservationTracking`, and transactions

The `@Observable` macro is where Observation becomes concrete. In the open-source macro implementation, the synthesized registrar is stored as `_$observationRegistrar`; the synthesized accessor helper `access(keyPath:)` forwards reads to that registrar; and the synthesized mutation helper `withMutation(keyPath:_:)` forwards writes through `ObservationRegistrar.withMutation`. For each observed property, the generated `get` accessor calls `access(keyPath:)`, the plain `set` accessor wraps assignment in `withMutation(keyPath:)`, and the `_modify` accessor calls `access`, `willSet`, and `didSet` around the yielded inout storage. Transactions enter the picture through `withMutation`. In `ObservationRegistrar`, `withMutation` calls `willSet` before the mutation closure and `didSet` afterward. ŅłĆciteŅłéturn15view0Ņłéturn9view3Ņłü

This also explains why SwiftUI does not use ObservationŌĆÖs `AsyncSequence` interfaces for core view invalidation. A later Swift Forums discussion spelled out a real consequence: if you re-arm observation on an asynchronous hop, you can miss the current animation transaction deadline. For unread changes, `@Observable` is where the evidence is strongest and the answer is closest to ŌĆ£yes, by design.ŌĆØ AppleŌĆÖs documentation for `withObservationTracking` says it tracks access to properties inside the closure and only informs you when one of those participating properties changes. There is a newer source-level refinement worth calling out because it affects the current state of the system in 2026. This means current Observation behavior is more aggressive about dropping vacuous setter notifications than the original Swift 5.9-era model, but that optimization is version-sensitive and does not apply uniformly to `_modify` mutation paths. ## How all of this fits into view evaluation and dirty marking

The highest-confidence public model is this: before a viewŌĆÖs `body` is evaluated, SwiftUI refreshes its dynamic-property boxes; during `body` evaluation, it collects dependencies; when underlying state mutates, it marks the right dependencies dirty; then it re-evaluates only the affected view scopes and reconciles the results. OpenSwiftUI shows this very explicitly for local state: `StatePropertyBox.update` returns a Boolean indicating whether the property should be treated as changed, and `GraphHost` turns graph invalidations into `graphDidChange()` or `setNeedsUpdate()` calls. That is a classic dirty-marking pipeline, not a brute-force full-tree redraw. ŅłĆciteŅłéturn38view1Ņłéturn37view4Ņłéturn38view3Ņłü

For Observation-backed models, the most plausible wiring is that SwiftUI evaluates a render scope inside its SPI overload of `withObservationTracking`. The source exposes those SwiftUI-only overloads, and objc.ioŌĆÖs source-level reimplementation describes the execution of a viewŌĆÖs `body` as being wrapped in `withObservationTracking` so every accessed key path becomes part of the dependency set. Apple has not published the exact SwiftUI code, so this should be read as a source-backed inference rather than a literal line-for-line statement from SwiftUI itself, but it fits the available evidence extremely well. ŅłĆciteŅłéturn11view0Ņłéturn27search2Ņłü

Tokamak helps illuminate the same pipeline from the opposite direction. Its `queueStorageUpdate` mutates stored state and then queues an update for the owning mounted element. The reconciler coalesces updates, schedules work, and later calls `mountedView.element.update(...)`. That is a simpler, coarser engine than SwiftUIŌĆÖs Observation-based path, but it still demonstrates the same phase separation: mutate storage, mark dirty, schedule reconciliation, then update the mounted node. ŅłĆciteŅłéturn35view0Ņłéturn35view1Ņłü

Once a view has been re-evaluated, the framework still has another chance to avoid work. FatbobmanŌĆÖs analysis of SwiftUIŌĆÖs update mechanism describes SwiftUI as building a fresh child-view instance when a parent updates and then quickly comparing old and new child instances to decide whether to recurse. TokamakŌĆÖs renderer guide makes the same distinction in different words: the reconciler decides what needs to mount, unmount, or update, and only then notifies the renderer. So ŌĆ£avoiding invalidation cascadesŌĆØ is only the first optimization layer; diffing and selective reconciliation are the next ones. ŅłĆciteŅłéturn26search17Ņłéturn29search4Ņłü

The practical result is that a state change can be cheap in several different ways. It can be ignored because nobody tracked the property. It can invalidate a view but still not force deep recursion because the child values compare equal enough. Or it can re-evaluate `body` without ultimately causing a platform redraw because the reconciler finds no renderer-visible change. SwiftUI performance depends on all three layers, not just on Observation. ŅłĆciteŅłéturn24search4Ņłéturn26search17Ņłéturn29search4Ņłü

## What the reimplementations reveal

OpenSwiftUI is the closest public stand-in for the ŌĆ£property read plus dirty graphŌĆØ model. Its state system has a per-location `wasRead` bit, a state signal, a `DynamicPropertyBox.update` method that can decline to mark a property changed if it was unread, and a `GraphHost` that converts graph invalidation into host updates. Tokamak is intentionally much simpler, and that simplicity is useful precisely because it shows what SwiftUI is trying to avoid. There is no equivalent of ObservationŌĆÖs access-list capture or per-property `wasRead` gating in the core path shown here. SwiftCrossUI lands somewhere in between. ## Practical conclusions

For basic `@State`, the strongest public evidence says SwiftUI-style engines can avoid invalidating a view for an unread local-state change, but Apple has not documented that guarantee in the same explicit terms it uses for `@Observable`. I would therefore treat unread `@State` elision as **likely and architecturally intended**, but not as a contractual behavior you should rely on across every OS version and every special case. The more defensible rule is that `@State` is local storage, and the views that actually read it are the ones that should drive update work. ŅłĆciteŅłéturn24search12Ņłéturn38view1Ņłü

For `@Binding`, there is no separate invalidation system to understand. A binding read contributes a dependency on the underlying location; a binding write mutates that location; any later invalidation comes from the source of truth and the consumers that read it. If you want to know whether a binding change will invalidate a view, the right question is not ŌĆ£does the binding track changes,ŌĆØ but ŌĆ£what storage is behind this binding, and who read it during the last update pass?ŌĆØ ŅłĆciteŅłéturn8view1Ņłéturn37view3Ņłéturn38view2Ņłü

For `@Observable`, the answer is much cleaner. The registrar records which key paths were accessed, `withObservationTracking` installs key-path-specific callbacks, and SwiftUI uses that synchronous observation path rather than async streams. So unread property changes are exactly what Observation is designed to avoid propagating. On current toolchains, equal/identical direct setter assignments may also be filtered before they become observation events at all. ŅłĆciteŅłéturn15view0Ņłéturn17view2Ņłéturn9view3Ņłéturn22view0Ņłéturn22view1Ņłü

If your goal is ŌĆ£avoid invalidation cascades,ŌĆØ the clearest engineering implications are to keep local ownership in `@State`, expose only the bindings that downstream views truly need, prefer `@Observable` over `ObservableObject` when property-level granularity matters, and avoid replacing framework-managed bindings with opaque closure bindings unless you actually need that flexibility. Those choices line up with both AppleŌĆÖs public direction and the behavior visible in the source-backed reimplementations. ŅłĆciteŅłéturn14search8Ņłéturn24search12Ņłéturn8view1Ņłéturn27search15Ņłü
