# SwiftTUI 0.1.0 Public-Release Readiness Audit

**Date:** 2026-06-08
**Scope:** All four public child repos (`swift-tui`, `swift-tui-web`,
`swift-tui-examples`, `swift-tui-site`), the public `SwiftTUI/.github` profile,
the live npm packages, and the release runbook — assessed for blockers to the
**first public `0.1.0` release and announcement**. Current org state: `0.0.18`.
**Severity:** Not announce-ready today. One true legal blocker (licensing) plus
a cluster of self-contradictions. ~1–2 focused days of reconciliation, not
feature work.
**Method:** 10-dimension parallel audit (docs, framework onboarding, site, WASM
host, examples, implementation maturity, DocC coverage, legal/hygiene,
overclaim risk, release/versioning), 59 raw findings, synthesized and ranked by
value-per-effort. Every blocker below was **independently re-verified against
live state** (npm registry, remote git tags, the public profile README, file
contents) — see the ✓ column.

---

## TL;DR

- **The product is well above a typical 0.1.0.** API surface is curated and
  drift-gated, `Sources` has zero TODO/FIXME, examples are real apps, the site
  has a live WASM browser demo, and the docs are unusually honest. The gap is
  **release hygiene that rotted out of sync with reality** (0.0.7 → 0.0.18
  drift), **not missing features**.
- **One true legal blocker:** 3 of 4 public repos — and the live npm tarballs —
  ship with **no LICENSE** despite claiming MIT. Currently all-rights-reserved.
- **The most damaging issues are self-contradictions** a skeptical reader
  catches in minutes: the documented install pin *excludes* 0.1.0; the READMEs
  say npm publishing is "being finalized" while the packages are *live*; the
  honesty doc `VISION-GAP.md` is a 404 behind five live links; the public
  profile badge reads `0.0.4`.
- **Recommended call:** announce 0.1.0 explicitly as the **first public alpha
  minor that is still source-breaking** — not an API-stability milestone.
- **Important context correction:** the `swift-tui-org` root repo (where this
  audit runs) is **PRIVATE** (`gh repo view` confirmed). Its coordination-style
  README is therefore *not* the public front door — the public faces are the
  `SwiftTUI/.github` profile README, `swift-tui/README.md`, and swifttui.sh.

## What is already strong (do **not** spend effort here)

- **Curated, gated public API.** ~4.7k public symbols, but zero pending-review
  entries and a `--check` drift gate (`Scripts/lib/repo_policy_checks.sh:162`);
  `package`-access internals (`ViewGraph`, `AnyStateSlot`, `PrimitiveView`) are
  verifiably excluded from the baseline. The size is deliberate, not accidental.
- **No tech-debt markers in `Sources`.** Zero TODO/FIXME/unimplemented/stub. All
  26 `fatalError`s are internal-invariant / primitive-view / builder-artifact
  guards — none reachable from normal authoring.
- **Examples are real apps**, not stubs: gallery (10.7k LOC), layouts (9.3k),
  gifeditor (7.2k), gitviz (3k), each with a quality README, plus a self-aware
  `swift-tui-examples/docs/EXAMPLE-COVERAGE.md` coverage matrix.
- **The site is genuinely strong**: real landing page + pitch, a `/pipeline/`
  deep-dive, a **live interactive WASM browser demo** (Conway's Life →
  wasm32-wasi via `@swifttui/web`), real DocC composition, and a real Cloudflare
  Pages deploy with file-count/size guards and COOP/COEP isolation.
- **Unusually honest docs.** Pervasive "pre-1.0, single-maintainer, AI-assisted,
  source-breaking may land" hedging. The overclaim risk is in **stale facts**,
  not marketing superlatives.

### Blocker fixes

- **B1** — Copy `swift-tui/LICENSE` verbatim to the three repo roots **and** into
  `swift-tui-web/packages/web/` + `packages/build/` so npm auto-bundles it
  (verify with `npm pack --dry-run`). Note in `swift-tui-examples` that `Vendor/`
  keeps its own license. Re-publish both npm packages (happens anyway at the
  0.1.0 bump). Add a License section to each repo README.
- **B2** — Make the 0.1.0 call (see [Strategy](#the-010-strategy-call)); bump the
  documented floor to `.upToNextMinor(from: "0.1.0")` everywhere
  (`swift-tui/README.md:65`, `Hero.astro:54`, `Quickstart.astro:76`) and the 16
  example `exact:` pins; ship a one-paragraph "what 0.1.0 means" + a short
  "what changed since 0.0.18" migration note.
- **B3** — `git push origin :refs/tags/v0.1.0` on `swift-tui-web` and
  `swift-tui-examples`. Add a `git ls-remote --tags` pre-flight to `RELEASE.md`.
- **B4** — Restore/commit `swift-tui/docs/VISION-GAP.md` (content exists in the
  site's `.build-docs-work` copy); fix its two stale lines (the `0.0.7 alpha`
  status → `0.1.0`; move "npm publishing" from *Not yet built* → *shipped*). Add
  `VISION-GAP*.md` to the `bump_version.sh` sweep so it can't rot again.
- **B5** — Capture real terminal screenshots/GIFs of the four apps (they build at
  `0.0.18`; gallery supports `--tab`, gitviz emits chart output) and replace the
  SVGs. If captures can't land in time, **remove the Showcase nav entry + page
  from 0.1.0** rather than ship placeholders. Reconcile the gitviz subcommand
  count (prose says 17, SVG says 13) when replacing.

## Highest-value changes — ranked by value-per-effort

Not strict blockers, but the biggest first-impression payoff per hour. This is
the core answer to "what are the highest-value changes before announcing."

1. **`very-high` · S — Lead with `npm install @swifttui/web @swifttui/build`
   everywhere.** npm is **live at 0.0.18** (verified), yet `swift-tui/README.md:175`
   says "while npm publishing is being finalized" and routes through tarball
   URLs, while the root README says the opposite. The project contradicts itself
   on its marquee feature. Demote tarball URLs to a secondary note.
2. **`very-high` · S — Fix the stale `status-0.0.4 alpha` badge** on the *public*
   `SwiftTUI/.github` profile README (verified live; everything else is 0.0.18).
   It is the literal first version string a visitor sees. Better: point it at the
   latest-release shield so it can't re-rot. Add the profile README to the bump
   sweep (it is not a submodule).
3. **`very-high` · S — One disambiguation sentence vs `rensbreur/SwiftTUI`** (the
   established project that owns the "SwiftTUI" search term). The top announce
   comment will be "how is this different?" — pre-empt it (multi-host: terminal +
   WASI browser + native SwiftUI embed; deterministic snapshot-testable pipeline).
4. **`very-high` · S — Onboarding cliff fixes.** Every example command is
   `swiftly run swift …` but swiftly is never introduced (first copy-paste →
   `command not found`); and the framework README never shows a complete,
   copy-pasteable `Package.swift`. Add a prerequisite note + plain `swift run`
   fallback + one ~15-line manifest (mirror `swift-tui-examples/argparse/Package.swift`).
5. **`high` · M — One hero GIF in the GitHub READMEs.** A terminal-UI framework
   with **zero embedded media** where the traffic actually lands. An unused
   `threehosts.asciinema` recording already exists — export to GIF/SVG, embed in
   the framework, profile, and npm-`packages/web` READMEs, add a "Try it live →
   https://swifttui.sh" link (currently only the DocC subpath is linked).
6. **`high` · M — CHANGELOG + real 0.1.0 release notes.** None exist anywhere;
   `RELEASE.md` creates releases with a literal `--notes "..."` placeholder. Add a
   Keep-a-Changelog `CHANGELOG.md` to `swift-tui` (covering the absorbed 0.0.17
   Canvas/CanvasContext refactor) and switch the runbook to `--notes-file`.
7. **`high` · S — `CODE_OF_CONDUCT.md` + minimal issue/PR templates** in
   `swift-tui`. GitHub Community Standards flags the gaps; templates make
   announce-day triage survivable for a solo maintainer. (No CoC in any repo;
   `.github` dirs contain only workflows.)
8. **`medium` · M — Publicly disclose the active `#12` SIGSEGV.** A
   non-deterministic main-thread memory-corruption crash in the core render loop,
   ASan/TSan-clean, root-cause unidentified (`swift-tui/docs/KNOWN-TEST-FLAKES.md:22,89`).
   Disclose it as a known issue in the README/VISION risk section + CHANGELOG.
   Disclosure is the announce-minimum; the actual fix is `L` and can defer.
9. **`medium` · L — DocC member-doc gap on flagship modifiers.** 229/297 (77%) of
   public `SwiftTUIViews` funcs lack `///`, including `.padding`/`.frame`/
   `.foregroundStyle`/`.background`/`.overlay`/`.onAppear`/`.onChange`/`.task`/
   `.bold`/`.lineLimit`/`.focused`. Plus 59 `public *Modifier` impl structs leak
   onto the surface uncurated. For a framework whose pitch is "you already know
   the API," a bare `.padding(_:)` DocC page undercuts the promise. **For 0.1.0,
   document the top ~25 modifiers (4–5 files) and demote the lowering-helper
   `*Modifier` structs to `package` (CLAUDE.md policy already says they stay
   package-only).** Defer the broader 24.5%→higher push to post-1.0.

## The 0.1.0 strategy call

`VISION.md` sets the milestone path as **0.x → 0.9.0 beta → 1.0.0**, and the
project explicitly reserves the right to make source-breaking changes every
minor. Therefore **0.1.0 is defensible only if announced as a still-breaking
alpha minor, not an API-stability milestone.** Recommendation: keep the "alpha"
qualifier; add a one-paragraph "what 0.1.0 means" note to `VISION.md` /
`DEVELOPMENT.md`; do **not** imply stability anywhere in the announce copy.
Committing ~4.7k public symbols to hard stability would be the wrong promise —
but a *source-breaking alpha minor* makes that surface size a non-issue.

## Cut line

**Do before announce** (≈1–2 days, mostly `S`): **B1–B5**, plus high-value
**1–4** (S-effort impression multipliers), plus **5–7** if time allows.

**Defer to 0.2.0** (announce *without* these, decisively):
- The full DocC member-doc sweep beyond the ~25 flagship modifiers; the global
  coverage push.
- The `SwiftTUICore` public-surface audit/demotion (~2,183 "infrastructure"
  symbols — document *why* they're public, or demote later).
- *Fixing* (vs disclosing) `#12` if a forced repro can't land in time.
- CHANGELOGs in the secondary repos; SECURITY/CONTRIBUTING parity across siblings.
- Per-example screenshots beyond the 4–5 flagship captures; sitemap/SEO polish.
- The "start here" beginner path and run-command-style consistency.
- Restoring an org-root CI gate.

**Do not block the announce on** perfect docs coverage or a root CI gate. The
legal, contradiction, and version-reconciliation fixes are what gate the launch.

## Top 5 moves (priority order)

1. **LICENSE** into web/examples/site repos + into `packages/web` & `packages/build`
   (npm auto-bundles); re-verify with `npm pack --dry-run`.
2. **Make 0.1.0 coherent in one pass:** delete the 2 orphan `v0.1.0` tags, bump the
   install pin to `0.1.0` (fixing the `.upToNextMinor` exclusion), run the scripted
   version bump across all repos + the stale `0.0.4` profile badge + 16 example
   manifests. Add a pre-bump guard asserting all repos share the current tag and the
   target strictly exceeds every existing tag (would have caught the 0.0.17 skew).
3. **Restore `VISION-GAP.md`** at HEAD and **fix the npm "being finalized"
   contradiction** across all READMEs (lead with `npm install`) — both are
   self-contradictions a skeptical reader catches in minutes.
4. **Replace the 4 placeholder showcase graphics** (or pull the page) and **add one
   hero GIF** — export the already-recorded `threehosts.asciinema` — to the
   framework, profile, and npm web READMEs.
5. **Onboarding basics:** swiftly prerequisite + `swift run` fallback, a complete
   copy-pasteable `Package.swift`, a real CHANGELOG + 0.1.0 release notes, and one
   `rensbreur` disambiguation sentence.
